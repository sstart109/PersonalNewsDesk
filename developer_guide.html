Architectural Design

The **IllumiLib PersonalNewsDesk** is engineered as a zero-infrastructure, client-side aggregator. It handles high-frequency, complex content harvesting entirely within the boundaries of the browser context. By discarding typical backend database architectures and API intermediary pipelines, the platform secures perfect user privacy, infinite direct scalability, and a deployment overhead footprint of exactly zero bytes.

Single-Page Client-Side Pipeline

The core lifecycle of PersonalNewsDesk operates inside a single runtime sandbox. Upon loading, the system boots a modular framework written in raw JS that parses locally stored configurations, coordinates parallel requests to multiple feed streams via CORS gateways, runs data normalization filters, executes clustering algorithms, and renders an interactive grid interface in real-time.

Core Architectural Decoupling

The application breaks down architectural responsibility into four isolated logical domains:

  • Configuration Layer: Governs default RSS feed listings, followed author preferences, appearance rules, custom theme definitions, and local storage state persistence.
  • Harvesting & Proxy Pipeline: Implements a resilient gateway selector that multiplexes fetch actions through fallback proxy lists to bypass restrictive browser CORS locks.
  • Natural Language Clustering Engine: Tokenizes story headlines in the client environment, filters noise words, and dynamically associates matching stories from disparate publishers into localized top-priority events.
  • Dynamic DOM Rendering Loop: Coordinates low-latency UI repaints, font scaling actions, custom theme injection variables, and user-facing modals without introducing heavy framework runtimes.

Design Choice Context: Decoupled client-side feed management was selected over centralized API processing to guarantee that user configurations, reading habits, and customized feed parameters remain local. This ensures the app can never be compromised by centralized server breaches or commercial usage audits.

Technology Suite

Technology Stack

To align with strict IllumiLib performance paradigms, the technology stack completely excludes complex web frameworks, runtime compilers, and commercial asset managers. This clean footprint ensures instantaneous load speeds, low memory bounds, and guaranteed execution longevity across diverse devices.

Core Engineering Footprint

Layer Technology Chosen Engineering Rationale
Markup Semantic HTML5 Guarantees complete compatibility with native web engines, iframe containment boxes (such as Google Sites widgets), and search indexing architectures.
Styling Vanilla CSS3 Custom Properties Bypasses CSS-in-JS overhead. Allows the application to execute micro-second theme swaps by altering DOM container variables dynamically.
Logic Engine Modular ES6+ JavaScript Utilizes standard web APIs (`DOMParser`, `Fetch`, `IntersectionObserver`) to minimize memory footprints while executing parsing logic in parallel.
Persistence Local Browser Storage Maintains offline persistence of feed listings, active/disabled statuses, customized colors, columns, and history without API authentication structures.

Why Frameworks Were Eliminated

During architectural exploration, platforms like React, Next.js, and Tailwind CSS were evaluated and explicitly rejected for the following key reasons:

  • Google Sites Container Constraints: Google Sites hosts custom embeds inside nested, sandbox-restricted `iframe` elements. Complex JavaScript frames or bundlers load slowly within these sandboxes and create runtime execution faults.
  • Zero Dependency Lifecycle: External frameworks introduce dependency degradation. By relying exclusively on native DOM API calls, the codebase remains perfectly functional decades after its compile date.
  • Strict Styling Isolation: Vanilla CSS allows pixel-perfect encapsulation of styles, protecting the interface layout from parent site inheritance bugs.

Engineering Specifications

Engineering Core

Understanding the inner mechanics of our parsing pipelines is vital for any engineer working to implement custom aggregators or add new feeds to the desk.

1. CORS Proxy Fallback Multiplexing

Browsers block cross-origin RSS requests by default due to strict Same-Origin Policies (SOP). PersonalNewsDesk circumvents this by passing raw feed URIs through an array of five public and semi-private CORS gateways in a recursive, self-healing loop. If a gateway returns a transport failure or HTTP lock, the parsing system automatically catches the exception and falls back to the next proxy:

const proxies = [
    { type: 'feed2json', url: 'https://feed2json.org/convert?url=' },
    { type: 'rss2json', url: 'https://api.rss2json.com/v1/api.json?rss_url=' },
    { type: 'corsproxy', url: 'https://corsproxy.io/?url=' },
    { type: 'codetabs', url: 'https://api.codetabs.com/v1/proxy?quest=' },
    { type: 'allorigins', url: 'https://api.allorigins.win/get?url=' }
];

2. Handling Ad-Blocker Failures

Commercial aggregators like `rss2json` are frequently flagged and blocked by aggressive tracking filters (e.g., uBlock Origin). To prevent the application from rendering blank columns or throwing loading errors, the parser detects parsing exceptions caused by network blocks and instantly routes requests to alternate proxies like `corsproxy.io` or Base64-encoded XML engines.

3. Dynamic local storage Migrations

When critical feed configurations or default source listings are modified (such as removing invalid sources like "The Gateway Pundit"), clients' browsers might continue rendering stale data stored in their local web caches. To trigger an automatic client-side purge without manual settings intervention, the application compares configuration versions:

let sources = JSON.parse(localStorage.getItem('pnd_sources'));
let sourcesVersion = localStorage.getItem('pnd_sources_version');

if (!sources || sourcesVersion !== '7') {
    // Dynamically clean structures and re-inject verified defaults
    sources = defaultSources.filter(src => src.id !== 'gatewaypundit');
    localStorage.setItem('pnd_sources_version', '7');
    localStorage.setItem('pnd_sources', JSON.stringify(sources));
}

4. Google News XML Fallbacks

Feeds generated by Google News queries feature unique parameters that break generic JSON converters. When processing Google XML, our parser uses a native browser `DOMParser` to extract raw elements, dynamically strips standard branding suffixes (e.g., matching site strings or corporate domain suffixes), and cleans HTML tags to yield readable headlines.

5. Scraping Non-RSS Layouts & Client-Only Proxies

Columns that do not offer standard RSS feeds (such as Townhall's political cartoons) must be retrieved by scraping the live HTML and parsing it via the DOM. When routing these requests to bypass cross-origin restrictions, certain CORS proxies (like `corsproxy.io`) block server-side environments (such as testing scripts) but fully allow client-side browser requests. To resolve this, our scraping pipeline uses a multiplexed proxy checklist, ensuring client-safe endpoints are prioritized and fallbacks are triggered dynamically if a server block occurs.

Compilation Protocols

Build & Bundle Flow

Because the app is designed to run seamlessly as an embedded module on Google Sites, the final production asset must exist as a **single, self-contained standalone HTML document** that houses all styling parameters, JS modules, and image assets inline.

The Dual-Asset Layout Model

The code is engineered using a modular, split-pane developer strategy. Styles are authored in `styles.css`, behaviors are handled inside `app.js` and `themes.js`, and local testing is handled on `index.html`. This keeps source control modifications clean, readable, and highly maintainable.

Bundling `bundled_for_google_sites.html`

To compile the modular sources into the standalone bundle, we inject the dependencies directly. The build process follows these steps:

  • HTML Skeleton: The layout structure of `index.html` is used as the foundational template.
  • Style Inlining: The link tag `<link rel="stylesheet" href="styles.css">` is replaced with a style tag containing the complete contents of `styles.css`.
  • Script Inlining: Script source tags are parsed, and the exact production logic of both `themes.js` and `app.js` is embedded directly into bottom-aligned script blocks.
  • Base64 Asset Encoding: UI images and favicons (like `favicon.png`) are converted into inline Base64 data strings to bypass file-path dependencies.

Note on Embedding: Google Sites enforces a sandboxed container on custom HTML embeds, blocking external relative resource access. This is why compiling to `bundled_for_google_sites.html` is an absolute architectural requirement for production deployment.

Deployment & Release

Release & Git Deploy

PersonalNewsDesk leverages **GitHub Pages** to serve its source code and static interfaces to users at `https://sstart109.github.io/PersonalNewsDesk/`. Staging, testing, and production deployment must proceed via Git in a strict sequence.

GitHub Pages Synchronization Flow

Because there is no separate compilation server or automated action runner executing branch merges on the remote, every local modification to the core styles or logic must be fully bundled and pushed directly by the developer to ensure visual updates are instantly served to the web.

Step-by-Step Release Protocol

A junior engineer must always execute the following commands in the workspace root when committing changes:

# Step 1: Ensure code compilation is fully mirrored in bundled files
# Step 2: Stage all updated files (app.js, styles.css, bundled_for_google_sites.html)
git add .

# Step 3: Commit with clear, atomic descriptive summaries
git commit -m "Configure news desk feeds: clean gatewaypundit, bump config version to 7"

# Step 4: Push to the main branch to trigger GitHub Pages static hosting updates
git push origin main

Critical Verification Rule: After pushing your updates, never assume deployment success based on local server states. You must open your mobile browser or private tab and directly test `https://sstart109.github.io/PersonalNewsDesk/` to confirm that the changes compile correctly and that all feed columns load without console exceptions.

Activating GitHub Pages (404 Troubleshooting)

If the repository is newly created, switched between private and public states, or has Pages deactivated, visiting the target URL will result in a standard GitHub Pages 404 error page. In these scenarios, the automatic build workflow (triggered by git pushes) will not run. To resolve this, Pages must be explicitly enabled either via the GitHub repository web settings or programmatically using the GitHub CLI:

# Enable GitHub Pages via the GitHub CLI API
gh api --method POST repos/sstart109/PersonalNewsDesk/pages -F "source[branch]=main" -F "source[path]=/"

OCR Capture Companion

The PersonalNewsDesk is a dedicated administrative dashboard and portal within the wider **IllumiLib Product Suite**. To understand how news harvesting coordinates with the wider company ecosystem, developers should familiarize themselves with our core OCR technologies.

IllumiLib Capture Desktop Utility

The **IllumiLib Capture** desktop application is a highly optimized companion tool designed to bridges physical volumes and the digital cloud. It captures live book pages via high-definition camera feeds, executes sub-pixel corrections, and routes processed layouts to our cloud SaaS OCR engine.

OCR Layout Processing Workflow

Our core text normalization engine operates as follows:

  • Hardware Capture: High-resolution optical arrays capture text geometries under irregular library lighting conditions (candlelight, direct lamps).
  • De-skewing & Layout Analysis: Identifies double-page splits, removes human fingers holding page margins, and straightens curved text lines.
  • Enterprise OCR Engine: Processes raw segments using state-of-the-art multi-lingual model classifiers, supporting historical character sets and non-Latin typography (such as Greek and Hebrew).
  • Document Synthesis: Converts OCR segments into fully searchable, AI-queryable corpuses. Every single generated response features real-time, verified page-number citations, allowing serious researchers to quickly verify details against physical books.
